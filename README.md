# dumpio

A small C++ library for reading the binary dump format produced by the
`dumpDotInterpolate` OpenFOAM utility. It is the data-loading half of a test
harness used to validate a standalone, FPGA-targeted reimplementation of
OpenFOAM's surface `dotInterpolate` operation against ground-truth output
generated by OpenFOAM itself.

## What this is

`dumpio` parses a packed binary file containing the inputs and reference output
of OpenFOAM's face-interpolation operation for a single mesh and time step. It
reads the file, validates its integrity, performs basic sanity checks on the
data, and (in the test-harness build) makes the parsed arrays available to a
standalone compute kernel for comparison against the reference.

It deliberately has **no dependency on OpenFOAM**. Once a dump file exists, every
downstream step вҖ” reading, validating, running the kernel, comparing вҖ” happens
in plain C++ with no OpenFOAM headers or libraries. This is what makes the
validation portable: the dump file is a self-contained artifact that can be
checked on any machine, by anyone, without an OpenFOAM installation.

## The bigger project

This library is one component of a project to accelerate computational
hotspots of OpenFOAM CFD solvers on FPGA hardware using High-Level Synthesis
(HLS). The target hardware is a Microchip PolarFire FPGA; the reference
workload is the `motorBike` tutorial (an unstructured, snappyHexMesh-generated
case representative of industrial external-aerodynamics CFD).

The first kernel chosen for acceleration is the surface interpolation
operation `dotInterpolate`, which computes the face flux

```
phi(face) = Sf(face) . interpolate(U)(face)
```

where `Sf` is the face area vector, `U` is the cell-centred velocity field, and
`interpolate` is linear (distance-weighted) interpolation from the two adjacent
cell centres to the face. This operation is a profiling hotspot because it is
invoked by every flux, divergence, and gradient evaluation in the solver.

The end-to-end pipeline the project follows:

1. **Profile** the OpenFOAM solver to identify hotspots (done вҖ” `dotInterpolate`
   and the Gauss-Seidel smoother were the leading candidates).
2. **Isolate** the hotspot into a standalone C/C++ kernel with plain
   pointer/array arguments and no OpenFOAM types вҖ” synthesisable by HLS.
3. **Validate** the kernel on the CPU against OpenFOAM's own output before any
   FPGA work. вҶҗ *this library supports this step*
4. **Synthesise** the validated kernel with HLS (C-simulation, synthesis,
   cosimulation).
5. **Integrate** the generated IP into an FPGA bitstream and offload the kernel
   from the host at runtime.

Validating on the CPU first (step 3) cleanly separates *algorithmic* correctness
from *HLS-induced* issues that appear later. If the kernel matches OpenFOAM here,
any discrepancy after synthesis is an HLS or hardware problem, not a logic bug.

the harness and kernel can use.

## Binary format

The format is a 14-byte packed header followed by the data arrays in a fixed
order. All values are little-endian (x86). The format version allows the reader
to reject incompatible files.

### Header (14 bytes, packed, no padding)

| Field           | Type      | Size | Notes                                  |
|-----------------|-----------|------|----------------------------------------|
| `magicNumber`   | uint32    | 4    | `0xD07F0A01` вҖ” identifies the format   |
| `formatVersion` | uint8     | 1    | currently `1`                          |
| `elementType`   | uint8     | 1    | `0` = FP64 (double), `1` = FP32 (float)|
| `nFaces`        | int32     | 4    | number of **internal** faces           |
| `nCells`        | int32     | 4    | number of cells                        |

### Data section (arrays, in this exact order)

| Array       | Element  | Count    | Meaning                                 |
|-------------|----------|----------|-----------------------------------------|
| `Sf_x`      | double   | nFaces   | face area vector, x component           |
| `Sf_y`      | double   | nFaces   | face area vector, y component           |
| `Sf_z`      | double   | nFaces   | face area vector, z component           |
| `lambda`    | double   | nFaces   | linear interpolation weights, in [0, 1] |
| `U_x`       | double   | nCells   | cell velocity, x component              |
| `U_y`       | double   | nCells   | cell velocity, y component              |
| `U_z`       | double   | nCells   | cell velocity, z component              |
| `owner`     | int32    | nFaces   | owner cell index of each face           |
| `neighbour` | int32    | nFaces   | neighbour cell index of each face       |
| `phi_ref`   | double   | nFaces   | OpenFOAM reference output (ground truth)|

Vector fields are stored in **structure-of-arrays (SoA)** layout вҖ” the three
components are separate contiguous arrays, not interleaved. This matches the
memory access pattern the HLS kernel expects (independent burst-friendly
streams).

Total file size (for `elementType = 0`, FP64):

```
14  +  (4 * nFaces + 1 * nFaces) * 8        // Sf x/y/z, lambda, phi_ref
    +  (3 * nCells) * 8                      // U x/y/z
    +  (2 * nFaces) * 4                      // owner, neighbour
```

A file-size check against this formula is the fastest integrity test.

## Integrity and sanity checks

On read, `dumpio` performs:

1. **Magic number check** вҖ” rejects files that are not in this format or are
   truncated/corrupt at the head.
2. **Per-array byte-count reporting** вҖ” prints bytes read for each array, so a
   short read (truncated file) is immediately visible.
3. **Domain invariants** on the parsed data:
    - all `lambda` values lie in `[0, 1]` (interpolation weights must be bounded)
    - all `owner` and `neighbour` indices lie in `[0, nCells)`
    - `owner[i] < neighbour[i]` for every internal face (OpenFOAM's
      upper-triangular face-ordering convention)

These invariants are cheap and catch the bugs that a size check alone cannot вҖ”
wrong offsets, type mismatches, or component transpositions.

## Building

`dumpio` is built as a library and linked by the test-harness executable.
The project uses CMake.

```bash
mkdir build && cd build
cmake ..
cmake --build .
```

Requirements:

- A C++ compiler supporting C++20 (uses `std::ranges` algorithms for the
  invariant checks).
- CMake 3.20 or newer.

No OpenFOAM, no external numerical libraries.

## Usage

The test harness takes a dump file path as its single argument:

```bash
./test_harness dot_interpolate_dump.bin
```

It reads and validates the file, runs the standalone kernel on the loaded
inputs, and compares the result against `phi_ref`, reporting maximum absolute
error, maximum relative error, and RMS error across all faces.

Generating a dump file (the producer side, requires OpenFOAM):

```bash
# inside a motorBike case directory that has been run to some time step
dumpDotInterpolate
# produces dot_interpolate_dump.bin in the case directory
```

## Scope and limitations

- **Internal faces only.** Boundary and coupled-patch faces are out of scope for
  this version; the kernel and dump cover the internal-face interpolation loop.
- **Linear interpolation only.** The reference output must be generated with the
  `linear` interpolation scheme (the `lambda` weights are linear). If the source
  case uses a different scheme for `interpolate(U)`, the kernel will not
  reproduce `phi_ref`.
- **FP64 currently.** The format supports an FP32 element type for future use,
  but the current dumps and kernel run in double precision.
- **Little-endian only.** No byte-swapping; intended for x86 hosts.
